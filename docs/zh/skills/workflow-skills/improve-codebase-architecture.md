# 改进代码库架构 (Improve Codebase Architecture)

面向“哪里耦合太紧、哪里该收口”的架构诊断技能。它会先找出高摩擦模块，再给出可迁移的接口和 RFC 草案。

## 适用场景

- 想改善架构、拆分耦合、收敛边界
- 想找 refactor seam、深模块机会或更安全的接口设计
- 想把架构评审结果整理成 RFC 或 issue 草案

## 工作流程

1. 先确定范围：有路径、子系统或主题就收窄在其中。
2. 扫描代码库，找出跨文件跳转、重复适配器、浅模块和隐藏编排。
3. 汇总 3-5 个最值得深挖的候选点，并说明耦合原因、测试影响和风险。
4. 选定一个候选后，再比较几种不同接口设计，并分析迁移形态。
5. 最后按模板写出 RFC 草案；只有用户明确要求时才创建 GitHub issue。

## 主要资源

- `references/deepening-guide.md`：分类依赖与 RFC 模板前必须先读
- 代码发现工具、测试覆盖情况和边界行为
- 本地生成的 `architecture-rfc-*.md` 草案

## 关键约束

- 不要为了“有产出”硬编 RFC；没有强候选就直接说没有
- 重点看行为边界，不要只盯着当前文件结构
- 默认先草拟，再考虑发布到 GitHub
- 边界测试比内部调用顺序更重要
