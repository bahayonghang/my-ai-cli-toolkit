# interview-plan

通过苏格拉底式访谈细化需求，直接调用原生 Plan 模式生成可执行计划。

## 概述

通过深入的苏格拉底式访谈，将模糊需求转化为清晰的工程规格，然后直接调用平台原生 Plan 模式生成可执行的实施计划。访谈结论无缝融入 Plan，无需管理中间文件。

## 使用场景

- 完善 `plan.md` 或其他需求文档
- 通过系统性提问细化模糊需求
- 直接生成可执行的实施计划
- 涵盖技术实现、UI/UX、边缘情况、风险评估
- 应用工程原则（KISS、YAGNI、DRY、SOLID）

## 触发词

- "帮我完善这个 plan"
- "对这个项目进行访谈"
- "interview-plan"
- 草稿需求需要全面细化时

## 工作流程

### 1. 文档分析

读取 `plan.md` 或用户指定的文档，分析：
- 核心业务目标和价值主张
- 现有技术方案的完整性
- 模糊或缺失的关键信息
- 潜在的技术债务和风险点

### 2. 系统性访谈

使用 `AskUserQuestion` 工具进行多维度深度访谈：

**A. 工程原则审查（必须）**
- KISS & YAGNI：该功能是否为 MVP 必需？是否可以简化？
- DRY：识别可抽象的模式
- SOLID：单一职责、可扩展接口、依赖管理

**B. 技术实现细节**
- 技术栈选型及理由
- 数据结构与算法
- 性能瓶颈与优化
- 第三方依赖与风险
- 测试策略

**C. API 设计（如适用）**
- API 清晰性与逻辑分组
- 请求/响应模型
- 认证与授权
- 错误处理与状态码

**D. UI/UX 设计（如适用）**
- 用户交互流程
- 边界情况 UI 反馈
- 响应式设计与可访问性

**E. 风险评估**
- 技术债务容忍度
- 安全性考量
- 可维护性 vs 速度权衡

**F. 边缘情况与异常处理**
- 网络故障、超时、重试
- 并发冲突与降级策略

### 3. 高风险操作检测

主动警告以下操作：
- 文件删除 / Git 强制推送
- 环境变量修改 / 数据库结构变更
- 批量依赖更新

### 4. 进入原生 Plan 模式

访谈完成后，直接调用 `EnterPlanMode` 进入原生 Plan 模式：
- 整理访谈结论写入计划文件
- 任务分解与优先级排序
- 依赖关系与执行顺序
- 每个任务的验收标准
- 风险缓解措施

用户审批计划后，调用 `ExitPlanMode` 进入执行阶段。

## 提问原则

1. **深度优先**：避免显而易见的问题，挖掘隐藏假设与风险
2. **结构化**：按工程原则 → 技术细节 → 风险权衡的顺序提问
3. **引导式**：帮助用户思考"为什么"而非仅仅"是什么"
4. **务实性**：关注可落地性，避免过度理论化
5. **上下文感知**：根据项目类型调整访谈重点（API/CLI/库/应用）

## 示例对话

**步骤 1：启动访谈**

> **用户**：帮我完善这个 plan
>
> **系统**：已读取 `plan.md`。该项目计划构建一个用户管理 API。在开始访谈前，请确认：这是一个面向公网的 API 还是内部服务？

**步骤 2：深度提问**

> **用户**：内部服务
>
> **系统**：明白。根据 KISS 原则，内部服务的认证可以更简单。推荐 JWT + Redis 黑名单方案，比完整 OAuth2 Server 节省 70% 开发时间。您倾向于哪种方案？

**步骤 3：进入 Plan 模式**

> **系统**：访谈完成！直接进入 Plan 模式生成可执行计划...
>
> *系统调用 `EnterPlanMode`，将访谈结论写入计划文件，用户审批后即可开始执行。*

## 最佳实践

- **准备充分**：在访谈前准备好业务上下文和约束条件
- **诚实回答**：对不确定的问题，可以要求提供多个选项及其权衡分析
- **敢于跳过**：遇到不相关的问题，直接指令"跳过，进入 Plan 模式"
- **无缝衔接**：访谈结论直接融入原生 Plan，无需手动管理中间文件
