# Kiro 集成命令

规格驱动开发工作流，支持迭代式需求、设计和任务生成，遵循 Kiro 的开发者友好方式。

## 命令

### `spec`

**描述**: 从粗略想法创建完整的功能规格，经历需求、设计和任务规划三个阶段，每个阶段都需要用户迭代审批。
**用法**: `/kiro:spec [功能名称或粗略想法]`

#### 工作流程

1. **需求收集** - 基于功能想法以 EARS 格式生成初始需求，包含用户故事和验收标准。输出到 `.kiro/specs/{feature_name}/requirements.md`
2. **用户审查（需求）** - 提示用户明确批准后再继续；根据反馈迭代修改
3. **设计文档** - 创建包含架构、组件、数据模型、错误处理和测试策略的全面设计。输出到 `.kiro/specs/{feature_name}/design.md`
4. **用户审查（设计）** - 提示用户明确批准；根据反馈迭代修改
5. **任务列表** - 生成编号复选框实施计划，面向 TDD 并引用需求。输出到 `.kiro/specs/{feature_name}/tasks.md`
6. **用户审查（任务）** - 提示用户明确批准；批准后工作流完成

#### 需求格式（EARS）

```markdown
### 需求 1

**用户故事:** 作为 [角色]，我想要 [功能]，以便 [收益]

#### 验收标准

1. WHEN [事件] THEN [系统] SHALL [响应]
2. IF [前置条件] THEN [系统] SHALL [响应]
```

#### 任务列表格式

```markdown
- [ ] 1. 设置项目结构和核心接口
    - 创建模型、服务、存储库的目录结构
    - 定义建立系统边界的接口
    - _Requirements: 1.1_

- [ ] 2. 实施数据模型
- [ ] 2.1 创建核心数据模型接口
    - 编写所有数据模型的 TypeScript 接口
    - _Requirements: 2.1, 3.3_
```

#### 关键规则

- 功能名称使用 `kebab-case`（如 `user-authentication`）
- 每个阶段需要用户明确批准后才能推进
- 任务仅限编码（不包含部署、UAT 或手动测试任务）
- 工作流仅创建规划工件——实际实施使用单独的命令

### `design`

**描述**: 创建包含研究和架构的全面功能设计文档。
**用法**: `/kiro:design [功能名称或粗略想法]`

#### 工作流程

1. **检查需求** - 确保 `.kiro/specs/{feature_name}/requirements.md` 存在
2. **研究** - 根据需求识别需要研究的领域；总结关键发现
3. **创建设计** - 生成包含必要章节的 `.kiro/specs/{feature_name}/design.md`
4. **用户审查** - 迭代直到获得明确批准

#### 设计文档章节

| 章节 | 内容 |
|------|------|
| 概述 | 功能摘要和目标 |
| 架构 | 系统架构和边界 |
| 组件和接口 | 组件定义及其契约 |
| 数据模型 | 数据结构和关系 |
| 错误处理 | 错误场景和恢复策略 |
| 测试策略 | 测试方法和覆盖计划 |

适当时包含 Mermaid 图表。设计决策包含理由说明。

### `task`

**描述**: 从已批准的功能设计生成实施任务列表。
**用法**: `/kiro:task [功能名称]`

#### 工作流程

1. **检查设计** - 确保 `.kiro/specs/{feature_name}/design.md` 存在
2. **生成任务** - 将设计转换为面向 TDD 的编号复选框列表
3. **用户审查** - 迭代直到获得明确批准

#### 任务约束

- 每个任务引用具体需求（如 `_Requirements: 1.1, 2.3_`）
- 任务限定为具体编码活动（编写、修改或测试代码）
- 增量推进——每个步骤基于前一步骤构建
- 最多两级层次结构（史诗级别加小数编号子任务）
- 排除非编码任务（部署、UAT、文档、性能指标）

### `execute`

**描述**: 执行 Kiro 规格中的具体任务，专注于实现。
**用法**: `/kiro:execute [功能名称] [任务描述或任务编号]`

#### 工作流程

1. **读取规格文件** - 从 `.kiro/specs/{feature_name}/` 加载 `requirements.md`、`design.md` 和 `tasks.md`
2. **识别任务** - 定位请求的任务（或建议下一个未完成的任务）
3. **实施** - 执行单个任务，根据引用的需求进行验证
4. **停止等待审查** - 完成任务后停止；不自动推进到下一个任务

#### 关键规则

- 开始前始终读取全部三个规格文件
- 一次只执行一个任务——未经用户请求不继续下一个
- 如果任务有子任务，从子任务开始
- 可以回答关于任务的问题而不执行它们

### `vibe`

**描述**: 使用 Kiro 轻松、面向开发者的方式提供快速开发协助。
**用法**: `/kiro:vibe [问题或疑问]`

轻量级模式，用于快速原型开发和直接问题解决。无规格工作流——只提供快速、实用的帮助。

#### 特点

- 最少步骤，直接沟通
- 回答问题、解释概念、提供代码片段
- 仅在用户建议时运行测试
- 优先并行调用工具以提高效率
- 仅在真正模糊时才澄清意图

## 示例

```bash
# 从粗略想法创建完整规格
/kiro:spec user-authentication

# 仅创建设计文档
/kiro:design payment-processing

# 从已批准的设计生成任务
/kiro:task payment-processing

# 执行具体任务
/kiro:execute user-authentication 2.1

# 快速开发帮助
/kiro:vibe "这个 API 的错误处理应该怎么设计？"
```

## 注意事项

- 完整规格工作流为：`spec`（需求 → 设计 → 任务）或通过 `design` / `task` 单独调用各阶段
- 所有工件存储在 `.kiro/specs/{feature_name}/` 下（requirements.md、design.md、tasks.md）
- 每个阶段需要用户明确批准后才能继续下一阶段
- `execute` 一次只实施一个任务
- `vibe` 独立于规格工作流——用于快速、非正式的协助
- 任务遵循 TDD 原则：测试驱动、增量推进、基于前一步骤构建
