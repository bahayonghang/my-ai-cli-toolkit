# 规格访谈技能 (Spec Interview)

## 概述

通过深入的苏格拉底式访谈，将草稿规格说明转化为完整、可执行的技术文档 (`spec.md` 或更新原有的 `plan.md`)。访谈完成后，系统将引导用户调用各类 AI 智能体的原生「Plan / 规划」模式，无缝衔接代码开发阶段，实现规范驱动开发（Spec-Driven Development）。

## 核心功能

- 🎯 系统性需求访谈与边界条件细化
- 🏗️ 工程原则严格审查（KISS、YAGNI、DRY、SOLID）
- ⚖️ 技术方案设计与多维度权衡分析
- 📝 自动生成标准化的技术规格书 (`spec.md`)
- 🤖 无缝对接当前主流 AI 工具的 Plan 工作流

## 工作流程

### 1. 文档读取与分析

读取 `plan.md` 或用户指定的任何需求草案，深入分析：
- 核心业务目标和核心价值主张
- 现有技术方案的完整度与可行性
- 模糊、矛盾或缺失的关键业务逻辑
- 潜在的技术债务、扩展性瓶颈和安全风险

### 2. 系统性访谈

使用 `AskUserQuestionTool` (或简单的系统提问) 进行多维度深度访谈：

#### A. 工程原则审查（必须）

**简单性检查（KISS & YAGNI）**
- 此功能当前是否为 MVP 绝对必需？能否延后至后续里程碑？
- 是否存在成本更低、更直接的替代方案能达到 80% 的业务目标？
- 当前抽象层次是否合理？是否在为“可能永远不会发生”的未来过度设计？

**重复性识别（DRY）**
- 识别相似逻辑模式，评估设计抽象通用组件的真实 ROI（投资回报率）
- 配置或业务模式是否会在项目中高频重复出现？
- 是否可以通过构建脚本、代码生成器或宏定义减少模板化骨架代码？

**架构健壮性（SOLID）**
- 模块/微服务职责是否高度内聚且单一？
- 接口与边界设计是否支持未来无痛扩展？
- 依赖关系网是否清晰且单向？是否存在环形或隐式依赖风险？

#### B. 技术实现细节

- 技术栈与第三方库选型依据与同类竞品对比
- 核心数据结构与算法选择的时间/空间复杂度评估
- 系统性能瓶颈点预测与针对性优化/削峰策略
- 第三方依赖项的必要性评估、开源协议（License）合规性与长期维护停更风险
- 自动化测试策略覆盖面（单元测试、API 集成测试、UI E2E、压测等）

#### C. API 与数据流设计（如适用）

- REST/GraphQL/gRPC API 命名的合理性与清晰度
- 资源路由设计与操作动词的精确语义匹配
- 请求负荷/响应 Schema 数据模型的完整性约束
- 身份认证（Authn）与权限授权（Authz）机制选型
- 全局一致的接口错误拦截处理与标准 HTTP 状态码映射设计
- 接口与客户端缓存策略、API 版本回退与向前兼容控制策略
- 敏感业务接口幂等性保障机制与高并发限流控制

#### D. UI/UX 体验设计（如适用）

- 用户核心交互流程链路的闭环完整性与心智模型一致性
- 关键节点的非理想状态反馈设计（全局/局部 Loading、Skeleton、空状态、网络异常兜底、降级展示）
- 断点与响应式布局策略（移动优先/桌面优先）、跨端兼容性基准线
- WCAG 系统性可访问性支持程度（键盘导航、ARIA 标签、高对比度）
- i18n 多语言环境与时区/货币本地化特殊处理方案

#### E. 风险前瞻与权衡妥协

- 业务上线期限与技术栈选型妥协造成的「技术债务」可接受阈值及其偿还计划表
- 数据隐私合规与系统安全风险拦截（防止 XSS/CSRF/SQLi、避免硬编码凭证、防越权访问等）
- 系统代码的可维护性与业务快速迭代验证速度之间的摇摆平衡考量
- 整体落地方案的物理资源成本预估模型及弹性伸缩策略
- 强依赖单一云厂商或三方 SaaS 的供应商单点锁定困惑应对

#### F. 边缘情况与系统健壮性

- 微服务调用重试熔断机制、防雪崩倒灌设计与合理的客户端超时配置
- 分布式并发更新冲突化解算法与多端数据最终一致性保障手段
- 主辅链路核心服务非致命异常优雅降级回退方案
- 关键业务行为结构化业务日志打点、监控告警水位设计策略
- 单节点崩溃或不可抗力导致的数据灾备转移、快速故障发现及业务恢复抢救预案

#### G. 落地工程化基础设施

- 环境部署、流水线及运维脚本的运行跨平台兼容性标准（POSIX CLI/PowerShell）
- 文档元数据补充规范及目录结构设计约束
- **核心落地策略**：基于各类 AI 工具能力的最佳 Plan 切片任务指导原则
- 统一的代码编码风格约定检查、自动化 Lint 及 Commit 信息书写规则约束
- Git 分支、PR Code Review、自动化发布上线全链路流程标准化方案定义

### 3. 高风险操作保护机制

如需求规格访谈中确认包含以下高风险破坏性行为操作，平台将主动拦截并发送风险醒目提醒：

```
⚠️ [高风险系统操作检测点] 喵～浮浮酱察觉到危险动作！
操作类型：[数据库删除 / 物理文件永久粉碎 / 凭证重置 / Git 风险推送覆盖 / 核心依赖基座替换 / 系统全局环境变量篡改]
高危影响面分析：[预估波及系统的核心业务模块，说明数据无法恢复等极端情况]
浮浮酱的工程建议防护措施：
  -  版本回退方案预填：[指导说明该如何快速通过冷备与系统快照进行回滚]
  -  灾备快照固化策略：[数据表层级/全量打包层级的导出策略要求及自动化归档设定]
  -  操作结果测试验收：[使用只读权限账号操作沙盒/UAT 测试环境与强制核验验证基准项]
(主人，这非常关键喵！如果出了问题就麻烦了，请您明确授权是否仍然执意进行？)
```

### 4. 生成统一实现规格书 (Spec) 与任务切片

完成全部系统性苏格拉底访谈对答后，系统将自动汇总讨论的所有知识细节，输出一份最终的 `spec.md` 或者将结论融合更新到用户原有的 `plan.md` 中。该文档是后续代码编写的基础宪章，格式包含但不限于：

- **业务背景与目标 (Context & Value)**
- **架构与核心设计决策 (Architecture & Design Decisions)**
- **API契约与数据模型 (API & Schema Contracts)**
- **任务里程碑拆解 (Task Breakdown)**：(兼容 AI Agent 工作流读取的精确指令集格式 `tasks.md` 逻辑)
- **风险评估及应对措施 (Risks & Mitigations)**

### 5. 无缝转接 AI Plan / Execution 模式

这是规格驱动落地的核心环节！当最终规格文档 `spec.md` 被确认保存后，助手将通过消息引导用户启动对应本地 AI 的 Plan 功能去阅读这份完善的指南并着手正式的迭代周期阶段：

> **浮浮酱的温馨提示**：规格访谈已结项并输出 `spec.md` 喵！(*^▽^*)
> 现在主人可以使用您顺手的工具执行后续的落地工作流啦：
> 
> - **使用 Claude Code / Codex CLI / Qwen CLI / Gemini CLI**:
>   请在终端运行：`[AI指令] -p "请阅读 spec.md 并执行开发"` 或直接唤醒 `\plan` 工具。
> - **使用 RooCode / Cline / Cursor 等插件工具**:
>   请切换至工具的 "Plan" (规划) 模式，并告知："请根据项目根目录的 spec.md 执行开发流"。
> - **如果继续在这个会话中使用浮浮酱**:
>   "好哒主人，浮浮酱已经准备就绪，请发令开始 Execution 执行吧喵！"

## 技术约束与环境限制

- **路径寻址**: 一律使用双修饰符包裹的字符串路径以及正斜杠 `/` 分隔符标准
- **高速全局搜索**: 硬性要求优先降级使用 Rust 编写的 `rg` (ripgrep) 组件，替代低性能正则搜索 `grep`
- **Git 操作安全防线**: **绝对禁止自动化执行**任何 `commit`, `push`, 分支衍合变基等版本控制写操作；必须将修改内容作为文本由用户自行确认。
- **上下文感知**: 必须严格遵循「先广度大范围读取项目骨架源码感知全貌 -> 后精确切片定位写逻辑修改行为 (Read-before-Write)」的编码开发思维原语。

## 访谈提问原则与提问艺术

1. **结构化递进**: 按 `业务目标(Why) → 工程原则(How Design) → 细节落地(How Impl) → 系统防护与妥协(What ifs)` 的金字塔剥洋葱式顺序层层深入提问。
2. **引导式反思深挖**: 扮演具备 10 年架构师经验的副手，不仅接收信息，更要抛出 "如果不这么做会怎样" 的反方验证观点以排查方案隐匿缺陷。
3. **消除模糊上下文**: 使用收敛性限定范围选项来代替开放式回答（如："这需要异步吗？A. MQ削峰 B. 协程并发 C. 无需异步"，而不是问"你怎么看"）
4. **精细上下文觉察**: 根据开发受众工程模式动态加权侧重：API/后端服务更侧重重试补偿与隔离，前端更考量交互体验渲染卡顿降级，命令行 CLI 侧重平台 POSIX 跨端管道操作限制适配。

## 典型场景使用对话流参考

**阶段 1: 扫描并初勘计划**
> **用户**: "帮我分析并完善根目录下的 user-auth-plan.md 的方案，使用 spec-interview"
> **系统**: "收到主人的指令喵！(๑•̀ㅂ•́)✧ 浮浮酱已阅读您的文档，发现该模块是 SSO 统一单点登录平台构建规划。启动前，我们需要确认关键的授权模型基础，采用哪种流控协议：OIDC? SAML 2.0? OAuth 2 PKCE 授权码模式？"

**阶段 2: 原理与技术边界追问交锋**
> **用户**: "使用 OAuth 2 的授权码，我们只对自家生态系统开放内部流转不接入外外部社交登录"
> **系统**: "了然喵！根据 KISS/YAGNI 原则，如果纯粹局限于内网鉴权互通，我们是否有必要搭建如此庞大完整的 OAuth2 Server？这需要预留 15 天的开发周期和复杂的 Client_ID 管理密钥对轮换基础设施...既然仅在内网集群流动穿透鉴权信息，或许基于非对称加密签名发放自包含状态的 JWT 并辅以 Redis 黑名单撤销集群控制更轻量级，您觉得牺牲一部分规范性换取 70% 投入人日减少可行吗喵？"

**阶段 3: 落锤定音与输出规格**
> **用户**: "有道理，用 JWT+Redis 驱逐方案足矣"
> **系统**: "好主意，浮浮酱为您点赞！(๑ˉ∀ˉ๑) ... [历经其他一系列如安全审计日志记录、降级备库探察的交锋]。
> **访谈收尾总结**: "已经全部理清喵～ 我为您更新了 `spec.md`。包含了我们在防范缓存击穿等高并发下 JWT 黑名单验证限流的具体做法和分步里程碑任务。
> 
> 现在已经可以呼叫 AI 干实事了：可以输入 `claude -p "读取 spec.md 开始你的敲代码之旅"` 推进下一步啦！" 

## 最佳实践心得

- **上下文预热**: 在触发该技能前最好先提供现有项目环境配置，例如依赖栈是什么语言/框架，系统已有中间件现状等，避免无效的「真空架构设想」。
- **选择性忽略**: 对于简单的低难度 CRUD 表单业务拼接不需要经历这套大动干戈的体系访谈重塑，果断要求 AI 快刀斩乱麻使用常规编写修改。
- **敢于拒绝提问**: 遭遇 AI 不断发散长篇大论的不关注偏门问题，直接霸道指令 "跳过后四个提问方向，按我说的主流程默认配置强行生成 `spec.md`"。
