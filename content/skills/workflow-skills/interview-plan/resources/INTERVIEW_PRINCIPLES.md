# 访谈提问原则

## 1. 结构化递进

按 `业务目标(Why) → 工程原则(How Design) → 细节落地(How Impl) → 系统防护与妥协(What ifs)` 的金字塔式顺序层层深入提问。

## 2. 引导式反思深挖

扮演具备 10 年架构师经验的副手，不仅接收信息，更要抛出 "如果不这么做会怎样" 的反方验证观点以排查方案隐匿缺陷。

## 3. 消除模糊上下文

使用收敛性限定范围选项来代替开放式回答。

**交互方式约束：必须通过 `AskUserQuestion` 工具呈现选项**，使用户可在 UI 中直接点击选择，禁止以纯文本 A/B/C 列表要求用户手动输入。每个问题提供 2-4 个 `options`，每个 option 包含简洁的 `label`（1-5 词）和解释性的 `description`。相互独立的问题合并到同一次调用（最多 4 个 `questions`）。

**示例**：
- ✅ `AskUserQuestion({ questions: [{ question: "异步策略选型？", header: "异步", options: [{ label: "MQ削峰", description: "..." }, { label: "协程并发", description: "..." }, { label: "无需异步", description: "..." }] }] })`
- ❌ "你怎么看"
- ❌ 纯文本输出 "A. MQ削峰 B. 协程并发 C. 无需异步" 要求用户手动回复

## 4. 精细上下文觉察

根据开发受众工程模式动态加权侧重：
- **API/后端服务**：侧重重试补偿与隔离
- **前端**：侧重交互体验渲染卡顿降级
- **CLI**：侧重平台 POSIX 跨端管道操作限制适配

## 5. 方案探索（头脑风暴集成）

在完成维度提问后、进入风险保护前，基于收集的信息提出 2-3 种可行实现方案，附带权衡分析和推荐。通过 `AskUserQuestion` 让用户选择方案，选定方案融入最终 Plan 的架构决策部分。详见 `resources/BRAINSTORMING_INTEGRATION.md` 的方案提议协议。
