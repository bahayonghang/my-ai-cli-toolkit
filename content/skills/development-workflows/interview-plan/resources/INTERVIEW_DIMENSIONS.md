# 访谈维度详表

## A. 工程原则审查（必须）

### 简单性检查（KISS & YAGNI）
- 此功能当前是否为 MVP 绝对必需？能否延后至后续里程碑？
- 是否存在成本更低、更直接的替代方案能达到 80% 的业务目标？
- 当前抽象层次是否合理？是否在为"可能永远不会发生"的未来过度设计？

### 重复性识别（DRY）
- 识别相似逻辑模式，评估设计抽象通用组件的真实 ROI
- 配置或业务模式是否会在项目中高频重复出现？
- 是否可以通过构建脚本、代码生成器或宏定义减少模板化骨架代码？

### 架构健壮性（SOLID）
- 模块/微服务职责是否高度内聚且单一？
- 接口与边界设计是否支持未来无痛扩展？
- 依赖关系网是否清晰且单向？是否存在环形或隐式依赖风险？

## B. 技术实现细节

- 技术栈与第三方库选型依据与同类竞品对比
- 核心数据结构与算法选择的时间/空间复杂度评估
- 系统性能瓶颈点预测与针对性优化/削峰策略
- 第三方依赖项的必要性评估、开源协议合规性与长期维护停更风险
- 自动化测试策略覆盖面（单元测试、API 集成测试、UI E2E、压测等）

## C. API 与数据流设计（如适用）

- REST/GraphQL/gRPC API 命名的合理性与清晰度
- 资源路由设计与操作动词的精确语义匹配
- 请求负荷/响应 Schema 数据模型的完整性约束
- 身份认证（Authn）与权限授权（Authz）机制选型
- 全局一致的接口错误拦截处理与标准 HTTP 状态码映射设计
- 接口与客户端缓存策略、API 版本回退与向前兼容控制策略
- 敏感业务接口幂等性保障机制与高并发限流控制

## D. UI/UX 体验设计（如适用）

- 用户核心交互流程链路的闭环完整性与心智模型一致性
- 关键节点的非理想状态反馈设计（Loading、Skeleton、空状态、网络异常兜底）
- 断点与响应式布局策略（移动优先/桌面优先）、跨端兼容性基准线
- WCAG 系统性可访问性支持程度（键盘导航、ARIA 标签、高对比度）
- i18n 多语言环境与时区/货币本地化特殊处理方案

## E. 风险前瞻与权衡妥协

- 业务上线期限与技术栈选型妥协造成的「技术债务」可接受阈值及偿还计划
- 数据隐私合规与系统安全风险拦截（XSS/CSRF/SQLi、硬编码凭证、越权访问）
- 系统可维护性与业务快速迭代速度之间的平衡考量
- 整体落地方案的资源成本预估模型及弹性伸缩策略
- 强依赖单一云厂商或三方 SaaS 的供应商锁定应对

## F. 边缘情况与系统健壮性

- 微服务调用重试熔断机制、防雪崩设计与客户端超时配置
- 分布式并发更新冲突化解算法与数据最终一致性保障
- 核心服务非致命异常优雅降级回退方案
- 关键业务行为结构化日志打点、监控告警水位设计
- 单节点崩溃数据灾备转移、快速故障发现及业务恢复预案

## G. 落地工程化基础设施

- 部署流水线及运维脚本的跨平台兼容性标准（POSIX/PowerShell）
- 文档元数据补充规范及目录结构设计约束
- 基于 AI 工具能力的最佳 Plan 切片任务指导原则
- 统一的代码风格约定、自动化 Lint 及 Commit 规则约束
- Git 分支、PR Code Review、自动化发布上线全链路流程标准化
