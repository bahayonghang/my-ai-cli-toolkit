# Execution Rules Specification

执行阶段的强制规则和约束条件。

## When to Use

- Phase 4 执行文件生成时必须遵守
- 任何涉及模板下载和文件写入的操作
- 变量替换和内容验证时参考

---

## 1. 模板下载规则 [MANDATORY]

### 强制步骤

在写入任何文件之前，必须完成：

1. **必须先读取** `specs/template-catalog.md` 获取精确文件路径
2. **必须执行** `git clone` 下载模板仓库到 `.gh-bootstrap-cache/`
3. **必须使用** `Read` 工具读取下载的模板文件
4. **只有在读取模板后**，才能基于模板内容生成配置

### 禁止行为

| 行为 | 状态 | 原因 |
|------|------|------|
| 跳过下载直接写文件 | ❌ 禁止 | 无法保证内容正确性 |
| 凭记忆/知识生成配置 | ❌ 禁止 | 模板可能已更新 |
| 不读取模板就开始 Write | ❌ 禁止 | 违反复制原则 |
| 不查阅 catalog 就下载 | ❌ 禁止 | 路径可能错误 |
| 猜测文件路径 | ❌ 禁止 | 必须使用 catalog 精确路径 |

---

## 2. 直接复制规则 [MANDATORY]

### 正确流程

```
1. Read 模板文件 → 获取完整内容
2. 原样复制内容 → 不做任何"理解"或"优化"
3. 仅替换变量 → {{projectName}}, node-version 等
4. Write 到目标 → 内容 = 模板原文 + 变量替换
```

### 禁止操作

```
❌ "理解模板意图"
❌ "根据最佳实践重新编写"
❌ "优化/简化/重构模板"
❌ "添加额外的步骤"
❌ "删除看起来不需要的步骤"
```

### 为什么必须直接复制

1. **GitHub 官方模板经过验证**: actions/starter-workflows 被数百万仓库使用
2. **自行编写容易出错**: 工作流语法复杂，细微错误会导致完全失效
3. **版本兼容性**: 官方模板使用最新的 Action 版本和最佳实践
4. **减少调试时间**: 直接复制的模板开箱即用

## 3. 变量替换规则

### 必须替换的变量

| 变量类型 | 占位符 | 替换为 | 来源 |
|---------|--------|--------|------|
| 项目名称 | `{{projectName}}` | 检测到的项目名 | Phase 1 |
| 项目描述 | `{{description}}` | 项目描述 | Phase 2 |
| 作者信息 | `{{author}}` | 用户提供的作者名 | Phase 2 |
| 年份 | `{{year}}` | 当前年份 | 自动获取 |
| 版本号 | `node-version: [18]` | 检测到的版本 | Phase 1 |
| GitHub 用户 | `{{owner}}` | GitHub 用户名/组织名 | Phase 2 |
| 仓库名称 | `{{repo}}` | 仓库名称 | Phase 2 |
| 组件特定 | `CODECOV_TOKEN` | Secret 名称 | Phase 2 |
| 赞助账号 | `github_sponsors: username` | 赞助平台账号 | Phase 2 |

### 绝对不要替换

以下是 GitHub Actions 语法，必须保持原样：

```yaml
${{ secrets.GITHUB_TOKEN }}  # 保持原样
${{ github.* }}              # 保持原样
${{ steps.* }}               # 保持原样
@v4                          # Action 版本保持原样
```

### 替换验证

替换完成后检查：
- [ ] 无遗留的 `{{placeholder}}` 占位符
- [ ] 所有 `${{ }}` 表达式保持不变
- [ ] Action 版本号未被修改

## 4. 文件操作规则

### 目录创建

```bash
# 必须创建的目录
mkdir -p .github/workflows
mkdir -p .github/ISSUE_TEMPLATE
mkdir -p .gh-bootstrap-cache  # 临时缓存
```

### 备份策略

```
[BACKUP RULE]
对于 plan.backup 中的文件：
1. 复制到 .bak 后缀
2. 记录备份结果
3. 备份失败则中止该文件的覆盖操作
```

### 清理策略

```
[CLEANUP RULE]
执行完成后：
1. 删除 .gh-bootstrap-cache/ 目录
2. 保留 .bak 备份文件供用户参考
3. 可选：用户可选择保留缓存以供调试
```

## 5. 错误处理规则

### 下载错误

```
[ON DOWNLOAD ERROR]
1. 记录下载失败的仓库和错误信息
2. 尝试使用内置默认模板（如有）
3. 通知用户哪些组件使用了默认配置
4. 继续处理其他组件
```

### 生成错误

```
[ON GENERATION ERROR]
1. 记录错误详情（文件、行号、原因）
2. 跳过该组件
3. 继续处理其他组件
4. 在最终报告中标记失败项
```

### 回滚策略

```
[ROLLBACK STRATEGY]
当发生严重错误时：
1. 删除本次新创建的所有文件
2. 从 .bak 恢复原文件
3. 清理临时文件
4. 报告回滚结果
```

## 6. 验证规则

### 文件验证

| 检查项 | 方法 | 失败处理 |
|--------|------|----------|
| 文件存在 | `ls` 检查 | 标记为失败 |
| 文件非空 | `size > 0` | 标记为失败 |
| YAML 语法 | 基本格式检查 | 警告但不阻断 |
| 变量替换 | 搜索 `{{` | 警告需人工确认 |

### 验证输出格式

```json
{
  "validation": {
    "passed": ["ci.yml", "bug_report.md"],
    "failed": [],
    "warnings": ["dependabot.yml: 建议检查更新频率"]
  }
}
```

## Quick Reference

### 执行前检查清单

- [ ] 已读取 `specs/template-catalog.md`
- [ ] 已创建必要目录结构
- [ ] 已完成需要备份的文件备份
- [ ] 已下载所需模板仓库

### 执行中检查清单

- [ ] 每个文件都从模板读取
- [ ] 仅执行变量替换，无内容重写
- [ ] 保留所有 GitHub Actions 语法
- [ ] 记录每个操作的结果

### 执行后检查清单

- [ ] 所有目标文件已创建
- [ ] 验证通过或警告已记录
- [ ] 临时文件已清理
- [ ] 执行结果已汇总
